Ready-to-run examples for the **SSL certificate check API** in six languages: **Python, TypeScript (Node.js), Go, Java, C#, PHP.**
Find out **when a domain's certificate expires**, who issued it, which names it covers (SAN), which TLS version the port negotiates and whether the chain of trust is intact. **SSL expiry monitoring** and **TLS certificate inspection** in one HTTP request — no `openssl s_client`, no TLS handshake of your own.

Every example **runs out of the box — no signup, no key, no card.** A public demo key is baked in.

```bash
git clone https://github.com/atlorium-api/ssl-certificate-check-api-client
cd ssl-certificate-check-api-client/python && pip install -r requirements.txt && python main.py
```

## The point: the example exits with a non-zero status code

This is not a JSON printer — it is **a ready-made check for cron and CI**. The program decides for itself whether things are fine and reports it through the exit code, so nothing needs to parse its output:

| Exit code | When |
|---|---|
| `0` | **OK** — every certificate is valid and expires in more than 30 days |
| `1` | **WARNING** — fewer than 30 days left: time to renew |
| `2` | **CRITICAL** — fewer than 7 days, invalid, self-signed, broken chain, or **host not covered by SAN** |
| `3` | The check could not be performed: API or network error |

So a GitHub Actions step is a one-liner — it turns the job red on its own:

```yaml
name: ssl-expiry

on:
  schedule:
    - cron: '0 7 * * *'   # every morning
  workflow_dispatch:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: '3.12' }
      - run: pip install -r requirements.txt
        working-directory: python
      # Non-zero exit = red job = e-mail notification.
      # No grepping the output, no external monitoring service.
      - run: python main.py example.com api.example.com www.example.com
        working-directory: python
        env:
          ATLORIUM_API_KEY: ${{ secrets.ATLORIUM_API_KEY }}
```

The same thing works in cron — it mails you whenever the command exits non-zero:

```cron
0 7 * * * cd /opt/ssl-check/python && python main.py example.com api.example.com
```

## One request for many hosts

Given more than one host, the examples use the **batch endpoint** `/api/Certificate/batch`. Up to **10 hosts are checked in a single call** — one billable request and one rate-limit slot instead of ten, with the server checking the hosts in parallel:

```bash
python main.py atlorium.com github.com api.stripe.com
```

## What it is for

An expired certificate is a public outage: the browser blocks the site with a red screen, the mobile app stops talking to your API, the payment webhook silently stops being delivered. And yet it is predictable months in advance — it only needs to be put under monitoring once.

The examples do not just print JSON — they **apply** it. Each ships a `monitorExpiry()` function that returns a verdict:

1. **Expiry traffic light.** `daysRemaining < 7` — CRITICAL, `< 30` — WARNING. (Let's Encrypt certificates live 90 days, which makes 30/7 the practical thresholds.)
2. **Validity and chain.** `isValid`, the contents of `errors[]` and errors inside `certificateChain[]`: expired, self-signed, revoked, issued by an unknown CA.
3. **SAN mismatch.** The requested host is matched against `subjectAlternativeNames` — **including wildcards** such as `*.example.com`. This is a separate, sneaky failure: a certificate can be perfectly valid yet issued for a different name, and the browser still shows `NET::ERR_CERT_COMMON_NAME_INVALID`. The classic case: a `*.example.com` certificate installed on the bare `example.com` — the wildcard covers exactly one subdomain level and **does not** cover the apex domain.

## Authentication

The key goes in the `Authorization` header:

```
Authorization: Bearer YOUR_KEY
```

| Key | Behaviour |
|-----|-----------|
| `ak_sandbox_demo_mockdata_v1` | **Demo key.** Public, shared by everyone. Returns mocks, charges nothing, needs no account. Responses are deterministic — the same host always yields the same result, so you can assert on them in tests. |
| Live key | A real TLS handshake against the host. Get one at [atlorium.com](https://atlorium.com) |

Switching to a live key requires **no code changes** — every example reads an environment variable:

```bash
export ATLORIUM_API_KEY="ak_your_live_key"
```

Every sandbox response carries the header `X-Atlorium-Sandbox: true`, so mock data can never be mistaken for real data.

## Endpoints

Base URL: `https://atlorium.com`

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/Certificate` | Certificate of a single host |
| `GET` | `/api/Certificate/url` | Certificate by URL — host and port are extracted automatically |
| `GET` | `/api/Certificate/batch` | **Batch check** of up to 10 hosts in one request |

### `GET /api/Certificate`

| Parameter | In | Type | Description |
|-----------|----|------|-------------|
| `host` | query | string | Domain name or IP: `example.com`, `www.example.com`, `192.168.1.1` |
| `port` | query | int | Port, defaults to `443`. Range 1–65535. Useful for mail: `993` (IMAPS), `995` (POP3S), `465` (SMTPS), `8443` (alternative HTTPS) |

### `GET /api/Certificate/url`

| Parameter | In | Type | Description |
|-----------|----|------|-------------|
| `url` | query | string | Full URL: `https://example.com`, `https://example.com:8443/api/v1`. Host and port are parsed for you — handy for URLs copied from the browser |

### `GET /api/Certificate/batch`

| Parameter | In | Type | Description |
|-----------|----|------|-------------|
| `hosts` | query | string | Comma-separated hosts: `a.com,b.com,c.com`. **Maximum 10** per request; spaces around commas are allowed |
| `port` | query | int | A single port for **all** hosts in the list, defaults to `443`. Need different ports — make separate requests |

## Response fields

| Field | Type | Meaning |
|-------|------|---------|
| `isValid` | bool | **The key field.** Whether the certificate passed full validation |
| `daysRemaining` | int | **Days until expiry.** Usually the reason this API gets called |
| `expirationDate` | datetime | Expiry date and time (UTC) |
| `validFrom` | datetime | Start of the validity period |
| `issuer` | string | Issuer DN: `CN=R3, O=Let's Encrypt, C=US` |
| `subject` | string | Subject DN: `CN=example.com` |
| `subjectAlternativeNames` | array | **SAN — the names the certificate covers.** This is what browsers match against, not CN. May contain wildcards: `*.example.com` |
| `errors` | array | Reasons for invalidity: expired, unknown CA, name mismatch, revoked |
| `tlsVersion` | string | Negotiated TLS version: `Tls13`, `Tls12` |
| `serialNumber` | string | Serial number |
| `thumbprint` | string | Certificate thumbprint |
| `signatureAlgorithm` | string | Signature algorithm, e.g. `sha256RSA` |
| `publicKeyAlgorithm` | string | Key algorithm: `RSA`, `ECDSA` |
| `keySize` | int | Key length in bits, e.g. `2048` |
| `isSelfSigned` | bool | Self-signed — public clients will reject it |
| `hasPrivateKey` | bool | Whether a private key is present (always `false` for a remote check) |
| `certificateChain` | array | Chain of trust: `{ subject, issuer, validFrom, validTo, thumbprint, isValid, errors }` |
| `connectionTime` | duration | How long the TLS connection took |
| `host` / `port` | string / int | What exactly was checked |
| `checkedAt` | datetime | Moment of the check (UTC) |

The batch endpoint returns an **array** of these objects, in the order the hosts were listed.

## Error handling

| Code | Cause | What to do |
|------|-------|------------|
| `400` | Host missing, port outside 1–65535, or more than 10 hosts in a batch | Check the parameters |
| `401` | Key missing, expired or invalid | Check the `Authorization` header |
| `402` | Insufficient credit balance | Top up at [atlorium.com](https://atlorium.com) |
| `429` | Rate limit exceeded | Retry with backoff — **but cap the wait** (see below) |
| `500` | Could not retrieve the certificate: host unreachable, timeout, no TLS on the port | Check host and port. **You are not charged for a failed check** |

All six examples map these codes to human-readable causes — see the `AtloriumError` class.

**About 429 and the wait cap.** Once the hourly quota is exhausted, the server honestly asks you to wait 40+ minutes. A client that blindly sleeps for as long as it is told hangs for that entire time (and burns the CI job budget). Hence `MAX_RETRY_DELAY = 120` seconds in every example: we never wait longer than the cap — we report an exhausted quota and exit with code `3`.

## FAQ

**How do I find out when a domain's certificate expires?** One GET request: `daysRemaining` tells you how many days are left, `expirationDate` gives the exact date. That field is what `monitorExpiry()` in the examples is built on.

**Why not just `openssl s_client -connect host:443`?** Because this needs no TLS stack in your code, no PEM parsing and no outbound connection to a third-party host from your own network — which a locked-down perimeter or a serverless function may simply not allow. The answer arrives as JSON, not as text to be scraped with regexes.

**What is a SAN mismatch and why does it matter?** Browsers match the hostname against `subjectAlternativeNames`, not `CN`. A certificate can be valid and unexpired yet not cover the name in question — and clients still get an error. The classic case: `*.example.com` does not cover `example.com` itself. The examples check this explicitly and report `MISMATCH`.

**Can I check ports other than 443?** Yes, via `port`: `993` for IMAPS, `995` for POP3S, `465` for SMTPS, `8443` for alternative HTTPS. Mail server certificates expire just the same — people simply notice it later.

**Does the demo key show my domain's real certificate?** No. The sandbox **generates** a plausible certificate: `isValid` is always `true`, `errors` is empty, SAN always contains the requested host, and `daysRemaining` lands between 30 and 365. As a result the WARNING, CRITICAL and MISMATCH branches practically never fire on the demo key — you will only see them with a live key against a genuinely broken host. In exchange the responses are deterministic and convenient to assert on in tests.
